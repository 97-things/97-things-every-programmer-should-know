# 对注释的评论

在我上大学的第一门编程课上，我的老师发给我们两张BASIC编码纸。黑板上的任务是“编写一个程序，输入并求解10个保龄球分数的平均值。”然后老师就离开了教室。这有多难呢？我不记得我最终的解决方案是什么，但我确定它里面有一个`FOR/NEXT`循环，并且总代码行数不会超过15行。编码纸——给你们正在阅读这篇文章的年轻人，是的，我们过去在实际输入到计算机之前要用手写的方式写代码——每张纸可以容纳大约70行代码。我对老师为什么会给我们两张纸感到非常困惑。由于我的字写得总是很糟糕，我用第二张纸重新整理了我的代码，希望能因为风格得到一些额外的分数。

令我惊讶的是，当我在下一堂课开始时收到作业时，我只得到了勉强及格的分数。（这对我在大学的余下时间来说是一个不祥之兆。）在我整洁地复制的代码的顶部，写着：“没有注释？”

仅仅知道程序应该做什么是不够的。这个任务的一部分是教会我，我的代码应该让下一个程序员能够理解。这是我一直没有忘记的教训。

注释并不邪恶。它们和基本的分支或循环结构一样在编程中是必需的。大多数现代语言都有类似于javadoc的工具，可以解析格式正确的注释以自动生成API文档。这是一个很好的开始，但远远不够。你的代码内部应该有关于代码应该做什么的解释。按照“如果写起来很难，那么阅读起来也应该很难”的格言来编码，对你的客户、雇主、同事和未来的自己都是不利的。

另一方面，你也可以在注释中过度。确保你的注释澄清了你的代码，而不是让它变得模糊不清。在你的代码中添加一些相关的注释，解释代码应该实现的目标。你的头部注释应该为任何程序员提供足够的信息，以便在不阅读代码的情况下使用你的代码，而你的行内注释应该帮助下一个开发人员修复或扩展它。

在一份工作中，我不同意上级做出的设计决策。作为一个年轻的程序员，我感到有些傲慢，于是我把指示我使用他们设计的电子邮件的文本粘贴到了文件的头部注释块中。结果，这家特定公司的经理们在提交代码时确实会审查代码。这是我第一次听说“职业发展受限的举动”这个词。
